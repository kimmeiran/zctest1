## 📋 重要开发规则

### 设计稿处理规则  
- 如果PRD中没有提供设计稿链接（Figma/Sketch等），则视为**没有设计稿**
- 开发时完全依据PRD文档中的文字描述、元素详细设计表和流程图进行开发
- 不需要主动询问或要求提供设计稿
- 如果PRD中明确提供了设计稿链接，则参考设计稿进行开发

### 图片资源与动效库的区别
- PRD中提到的"3D微芯片或电路板渲染图"、"缩略图"、"预览图像"等，属于**静态图片资源**
- 这些图片应使用 `<img>` 标签直接展示，**不需要使用动效库**（如Three.js、WebGL等）
- 动效库仅用于需要**动态交互**的3D场景或动画效果，不用于静态图片展示
- 如果PRD中明确说明是"预览图像"、"缩略图"、"渲染图"、"静态图片"等，一律视为静态图片资源

### 图片显示规范（后端返回的图片路径）
- **统一方式**：所有后端返回的图片路径，前端显示时**必须**使用 `getFullImageUrl(img.url)`（来自 `@/api/imageUpload.js`），无需额外“取数据”或调用下载接口。
- **使用示例**：
  ```html
  <img :src="getFullImageUrl(img.url)" />
  ```
  ```js
  import { getFullImageUrl } from '@/api/imageUpload.js'
  ```
- **适用范围**：
  - `questionContentImages`：题目中间的图片（content 中 `[img]` 占位符）
  - `attachedImage`：题目下方的图片（如「补画下图的三视图」中的图）
  - 导入题目、试卷预览、题库查看题目等所有涉及后端图片路径的场景
- **路径格式**：后端可能返回 `/exam/api/image/xxx.png`、`/exam/api/images/xxx/yyy.png` 或纯文件名，`getFullImageUrl` 均可处理。

### 联动选择（级联选择）规则
- 当存在**联动选择**（如省市区、分类-子分类、上级-下级等）时，**上级选项变更后，必须清空其下级/依赖的联动项**
- 例如：切换「省份」后，「城市」「区县」应置空并重新加载可选数据；切换「一级分类」后，「二级分类」应置空
- 实现时在上级字段的 `change`/`update` 事件中，将依赖该字段的所有下游选择框的值设为 `null` 或空，并视需求重新请求选项数据
- 避免用户界面出现「上级已改、下级仍为旧值」的不一致状态

### 必选项未选择时的提示规则
- 当用户**未选择必填项**（如下拉、单选框、级联选择等）时，应给出**友好提示**，引导用户完成选择，而不是直接禁用或使用生硬文案（如「不要选择」「禁止操作」等）。
- 推荐做法：在提交或失焦时校验，通过 **Message / Toast 提示**（如「请先选择 XX」）、或在该表单项下方显示 **inline 错误提示**（如「请选择省份」），并可选高亮该表单项。
- 避免：仅禁用按钮不说明原因、弹窗只写「不要选择」、无任何提示就阻止操作。应让用户明确知道「需要先选什么」。

### 试卷用途/规则类型（ruleType）统一规范
- **适用场景**：凡涉及「职业认定、竞赛、模拟考核」选择的模块（如组卷管理、组卷规则、自动组卷、手动组卷等），**必须**使用字段 **`ruleType`**，采用**数值**表示，不得使用字符串或其它字段名（如 `paperPurpose`）作为业务取值。
- **取值约定**：
  | ruleType | 含义     | 界面显示 |
  |----------|----------|----------|
  | 0        | 职业认定 | 职业认定 |
  | 1        | 竞赛     | 竞赛     |
  | 2        | 模拟考核 | 模拟考核 |
- **前端实现**：
  - 下拉选项使用数值：`<option :value="0">职业认定</option>`、`<option :value="1">竞赛</option>`、`<option :value="2">模拟考核</option>`，绑定使用 `v-model.number`。
  - 提交/查询接口参数统一传 **`ruleType`**（0/1/2）。
  - 列表/详情展示时，若后端返回 `ruleType` 或历史字段（如 `paperPurpose`），需用统一方法格式化为中文显示（如 `formatRuleType`）。
- **兼容说明**：若后端或历史数据返回字符串（如「职业认定」「竞赛」「CAREER_CERTIFICATION」），前端在**接收编辑回显**时可做一次映射为 0/1/2，再参与表单与提交；**新数据一律以 ruleType 数值与后端约定一致**。
- **涉及页面/组件**：`AutoPaperModal.vue`、`ManualPaperPage.vue`、`PaperManagement2.vue`、`QuestionGroupRule.vue`、`RuleDialog.vue` 等；后续新增模块凡包含「职业认定/竞赛/模拟考核」选择，均须参考本规则。

#### 1.1.1 组件库使用规范

**Web 端项目【允许并推荐】使用组件库**。

**核心理解**：

✅ **可以使用 Element Plus 组件但需定制样式**，含义如下：

- **可以使用组件库的功能**：Element Plus 提供的表单、表格、弹窗、导航等功能都可以使用
- **必须定制视觉样式**：组件库的默认视觉样式不能直接使用，需要根据项目风格库文档进行定制
- **功能与样式分离**：使用组件库的功能特性（如数据绑定、事件处理、API方法等），但视觉样式必须符合项目风格库

**组件库使用约定**：

1. **Element Plus 作为主要 UI 组件库**
   - 可正常使用其组件能力（表单、表格、弹窗、导航等）
   - 优先使用 Element Plus 提供的组件
   - 充分利用组件库的功能特性，提高开发效率

2. **组件样式定制要求**
   - 组件库的默认样式需根据项目设计文档进行覆盖或定制
   - 通过主题配置、CSS 变量或样式覆盖实现视觉风格统一
   - 确保组件视觉风格符合项目风格库文档要求
   - **详细样式定制规范请参考 [前端设计文档.md](./前端设计文档.md)**

3. **禁止事项**
   - ❌ 禁止直接使用组件库的默认视觉风格作为最终页面样式
   - ❌ 禁止组件视觉风格与项目风格库不一致
   - ❌ 禁止组件样式与页面整体风格不协调

**Element Plus 国际化配置**：

**必须配置**：Element Plus 默认使用英文，需要配置为中文。

**配置方法**（在 `main.ts` 中）：

```typescript
// main.ts
import { createApp } from 'vue'
import ElementPlus from 'element-plus'
import zhCn from 'element-plus/dist/locale/zh-cn.mjs' // 导入中文语言包
import 'element-plus/dist/index.css'
import App from './App.vue'

const app = createApp(App)

// 配置 Element Plus 使用中文
app.use(ElementPlus, {
  locale: zhCn, // 设置语言为中文
})

app.mount('#app')
```

**配置效果**：
- ✅ 日期时间选择器（DatePicker、TimePicker、DateTimePicker）显示中文
- ✅ 分页组件（Pagination）显示中文
- ✅ 表格排序、筛选等显示中文
- ✅ 所有组件内置文本显示中文

**注意事项**：
- 必须在 `main.ts` 中配置，确保全局生效
- 如果使用按需导入，需要确保导入中文语言包
- 配置后需要重启开发服务器才能生效

**常见问题与解决方案**：

1. **图标导入错误：图标不存在**
   - **错误信息**：`SyntaxError: The requested module does not provide an export named 'Building'`
   - **原因**：图标名称错误或该图标不存在于 Element Plus Icons 中
   - **解决方案**：
     - **检查图标名称**：Element Plus Icons 使用 PascalCase 命名，确保图标名称正确
     - **查看可用图标**：访问 [Element Plus Icons 官方文档](https://element-plus.org/zh-CN/component/icon.html) 查看所有可用图标
     - **常见图标名称对照**：
       - ❌ `Building` → ✅ 应为 `OfficeBuilding` 或 `House`
       - ❌ `Home` → ✅ 应为 `House`
       - ✅ `User` → 正确
       - ✅ `Setting` → 正确
       - ✅ `Edit` → 正确
       - ✅ `Delete` → 正确

2. **Vite 构建错误：图标模块未找到**
   - **错误信息**：`The requested module '/node_modules/.vite/deps/@element-plus_icons-vue.js' does not provide an export named 'XXX'`
   - **原因**：Vite 缓存问题或图标名称错误
   - **解决方案**：
     - **清除 Vite 缓存**（优先尝试）：
       ```bash
       # 删除 .vite 缓存目录
       rm -rf node_modules/.vite
       # Windows PowerShell
       Remove-Item -Recurse -Force node_modules\.vite
       # Windows CMD
       rmdir /s /q node_modules\.vite
       ```
     - **重新安装依赖**（如果清除缓存无效）：
       ```bash
       rm -rf node_modules
       npm install
       ```
     - **重启开发服务器**：清除缓存后重启 Vite 开发服务器
     - **检查图标名称**：确认图标名称是否正确，参考 Element Plus Icons 官方文档

#### 1.1.3 路由管理规范

#### 1.1.4 HTTP请求库规范

**Axios 版本**：Axios 1.9.0

**Axios 使用要求**：
- API接口文件统一放在 `src/api/` 目录
- 按业务模块划分文件（如 `user.ts`、`exam.js` 等）
- 支持混合使用 `.js` 和 `.ts` 文件
- 统一错误处理机制，提供友好的错误提示

**API 请求配置**：

```typescript
// api/user.ts
import axios from "axios"

// 使用Vite环境变量
// 开发环境
const API_BASE_URL = import.meta.env.VITE_API_BASE_URL || "/exam"
// 生产环境
// const API_BASE_URL = import.meta.env.VITE_API_BASE_URL || "/exam/api"

// 确保axios使用相对路径
axios.defaults.baseURL = ''

// API函数示例
export const getUserList = async (params): Promise<Response> => {
  try {
    const response = await axios.post(`${API_BASE_URL}/api/endpoint`, params, {
      headers: {
        "Content-Type": "application/json",
        Authorization: `${localStorage.getItem("token")}`,
      },
    })
    return response.data
  } catch (error) {
    console.error("操作失败:", error)
    throw error
  }
}
```

**统一请求头**：
- Token存储在`localStorage`：`localStorage.getItem("token")`
- 每次请求在Header中携带：`Authorization: ${localStorage.getItem("token")}`
- 注意：Authorization值前可能有空格，需根据实际情况调整

**响应格式**：
- 统一响应格式：`{ code: number, msg: string, data?: any }`
- `code === 200` 表示成功
- 分页数据格式：`{ rows: [], total: number }` 或 `{ list: [], total: number }`

### 2.2 TypeScript 类型规范

#### 2.2.1 `any` 类型使用规范

**禁止滥用 `any` 类型**，仅在以下情况允许使用：

- **第三方库类型缺失**：第三方库没有类型定义时，可使用 `any`，但必须添加注释说明
  ```typescript
  // 第三方库缺少类型定义
  // eslint-disable-next-line @typescript-eslint/no-explicit-any
  const lib: any = require('some-lib')
  ```

- **动态类型场景**：确实无法确定类型时，优先使用 `unknown`，其次考虑 `any`
  ```typescript
  // 优先使用 unknown
  function handleData(data: unknown) {
    if (typeof data === 'string') {
      // 类型守卫后使用
    }
  }
  ```

- **迁移过渡期**：从 JavaScript 迁移到 TypeScript 的过渡期，但必须标记 TODO
  ```typescript
  // TODO: 需要定义具体类型
  // eslint-disable-next-line @typescript-eslint/no-explicit-any
  const legacyData: any = {}
  ```

**禁止使用 `any` 的情况**：
- ❌ 为了快速开发而使用 `any`
- ❌ 可以定义类型但懒得定义
- ❌ 函数参数和返回值使用 `any`

#### 2.2.2 类型定义规范

- **优先使用 `interface`** 定义对象类型
- **使用 `type`** 定义联合类型、工具类型
- **明确函数参数和返回值类型**
- **使用泛型提高代码复用性**

```typescript
// ✅ 正确：使用 interface
interface User {
  id: string
  name: string
```

### 2.7 Vue 3 Composition API 开发规范

#### 2.7.1 函数定义顺序规范

**问题**：使用 `watch` 的 `immediate: true` 时，如果调用的函数定义在 `watch` 之后，会导致 "Cannot access before initialization" 错误。

**规范**：
- ✅ **正确**：先定义函数，再使用 `watch`

```typescript
// 1. 先定义所有函数
const resetForm = () => {
  // ...
}

// 2. 再使用 watch（可以安全调用 resetForm）
watch(
  () => props.data,
  (data) => {
    if (data) {
      // ...
    } else {
      resetForm() // ✅ 安全调用
    }
  },
  { immediate: true } // ⚠️ 使用 immediate: true 时必须确保函数已定义
)
```

- ❌ **错误**：函数定义在 watch 之后

```typescript
watch(
  () => props.data,
  (data) => {
    resetForm() // ❌ 错误：resetForm 还未定义
  },
  { immediate: true }
)

const resetForm = () => {
  // ...
}
```

**推荐顺序**：
1. Props/Emits 定义
2. Refs/Reactive 状态
3. Computed 属性
4. **函数定义**（resetForm, handleXXX 等）
5. Watch/WatchEffect
6. 生命周期钩子（onMounted 等）

#### 2.7.2 Watch 回调错误处理规范

**问题**：Watch 回调中如果出现错误，会导致整个组件崩溃。

**规范**：
```typescript
watch(
  () => props.data,
  (data) => {
    try {
      // 业务逻辑
      if (data) {
        form.name = data.name || '' // ✅ 使用空值合并
      } else {
        resetForm()
      }
    } catch (error) {
      console.error('Error in watch callback:', error)
      resetForm() // 错误时重置到安全状态
    }
  },
  { immediate: true, deep: false } // ✅ 明确指定 deep: false（除非需要深度监听）
)
```

#### 2.7.3 空值处理规范

**问题**：直接访问可能为 undefined 的属性会导致错误。

**规范**：
```typescript
// ❌ 错误：直接访问可能为 undefined 的属性
form.name = data.name
form.items = data.items

// ✅ 正确：使用空值合并运算符
form.name = data.name || ''
form.items = data.items || []
form.count = data.count ?? 0 // 使用 ?? 保留 0 值
```

#### 2.7.4 类型安全规范

**问题**：类型定义不完整导致运行时错误。

**规范**：
```typescript
// ✅ 定义完整的类型
interface FormData {
  name: string
  items: string[]
  count?: number // 可选属性明确标记
}

// ✅ 使用类型断言时添加检查
const data = response.data as FormData
if (data && data.name) {
  // 安全使用
}
```

### 6.2 components / views / api 划分规范

#### 6.2.1 components（组件）

**components/** 目录存放可复用的 Vue 组件：

- **通用组件**：可在任何项目中使用
  - PageHeader、InputField、SelectField、TextareaField 等
- **组件命名**：使用 PascalCase：`PageHeader.vue`、`InputField.vue`

#### 6.2.2 views（页面）

**views/** 目录存放路由页面组件：

- 每个路由对应一个页面组件
- 页面组件通常只负责布局和数据组装，具体功能由子组件实现
- 页面组件命名使用 PascalCase 或 kebab-case

```
views/
├── account/
│   └── index.vue        # 账号管理页
├── exam/
│   └── ExamManagement.vue  # 考试管理页
└── question/
    └── QuestionList.vue    # 题目列表页
```

#### 6.2.3 api（API接口）

**api/** 目录存放 API 调用：

- **按业务模块划分**：每个业务模块一个文件
- **支持混合使用**：可使用 `.js` 或 `.ts` 文件

```
api/
├── account.js          # 账号相关 API
├── user.ts            # 用户相关 API
├── exam.js            # 考试相关 API
└── questionBank.js    # 题库相关 API
```

### 6.3 公共逻辑存放规范

#### 6.3.1 utils（工具函数）

**utils/** 目录存放纯函数工具：

- 不依赖 Vue 的纯函数
- 可测试、可复用
- 按功能分类组织

```typescript
// utils/validators.js
export function validateRequired(value, fieldName) {
  if (!value || value.trim() === '') {
    return `${fieldName}不能为空`
  }
  return null
}

export function validateEmail(email) {
  const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/
  if (!emailRegex.test(email)) {
    return '邮箱格式不正确'
  }
  return null
}
```

#### 6.3.2 types（类型定义）

**types/** 目录存放 TypeScript 类型定义：

- 接口类型定义
- 可为第三方库补充类型定义

```typescript
// types/element-plus.d.ts
import { ElMessage } from 'element-plus'

declare module 'element-plus' {
  export { ElMessage }
}
```

## 八、提交流程

### 8.1 提交前必须通过的检查

#### 8.1.1 代码检查清单

提交代码前必须确保：

- [ ] **ESLint 检查通过**：无语法错误和规范错误
- [ ] **TypeScript 类型检查通过**：无类型错误（运行 `npm run build`）
- [ ] **代码格式统一**：代码格式符合项目规范
- [ ] **构建成功**：`npm run build` 无错误

#### 8.1.2 自动检查机制

**构建时检测**：
- 运行 `npm run build` 时会执行 `vue-tsc` 进行类型检查
- 确保所有TypeScript错误都已修复后再提交代码

**检查失败处理**：
- 如果检查失败，需要修复问题后重新提交
- 如果确实需要临时忽略某些类型错误，可以使用 `// @ts-ignore` 注释（不推荐）

### 8.2 Commit Message 规范

#### 8.2.1 格式规范

使用 [Conventional Commits](https://www.conventionalcommits.org/) 规范：

```
<type>(<scope>): <subject>

<body>

<footer>
```

**类型（type）**：
- `feat`：新功能
- `fix`：修复 bug
- `docs`：文档更新
- `style`：代码格式调整（不影响功能）
- `refactor`：代码重构
- `perf`：性能优化
- `test`：测试相关
- `chore`：构建工具或辅助工具变动

**示例**：
```
feat(exam): 添加考试管理页面

- 实现考试列表展示
- 添加搜索和筛选功能
- 添加分页功能
```

#### 8.2.2 简短提交

对于简单修改，可以使用简短格式：

```
feat: 添加登录功能
fix: 修复按钮点击无效的问题
docs: 更新 README
```

## 九、开发流程规范

### 9.0 项目初始化与启动

**项目初始化时必须自动创建以下文件**：

1. **根目录批处理文件**：
   - `start.bat`：启动开发服务器的批处理文件
   - `build.bat`：构建生产版本的批处理文件
   - 文件内容见 [6.1.1 根目录批处理文件](#611-根目录批处理文件) 章节

2. **项目配置文件**：
   - `package.json`：项目依赖和脚本配置
   - `vite.config.ts`：Vite 构建配置
   - `tsconfig.json`：TypeScript 配置
   - `.gitignore`：Git 版本控制忽略文件
   - `.env.development`：开发环境变量
   - `.env.production`：生产环境变量
   - `README.md`：项目说明文档

**项目初始化完成后必须自动执行**：

1. **安装依赖**：自动执行 `npm install`（或 `yarn install` / `pnpm install`）安装项目依赖
2. **启动开发服务器**：自动执行 `npm run dev`（或 `yarn dev` / `pnpm dev`）启动开发服务器
3. **验证启动**：确认开发服务器成功启动，可以正常访问

**注意**：
- 项目结构初始化完成后，应立即执行上述命令，无需等待用户确认
- 如果依赖安装或启动过程中出现错误，需要先解决错误再继续开发
- 开发服务器启动成功后，应告知用户访问地址（通常是 `http://localhost:3001`）
- 批处理文件创建后，用户可以通过双击 `start.bat` 快速启动开发服务器，或双击 `build.bat` 快速构建生产版本

### 9.1 逐页开发确认流程

**开发原则**：采用"完成一个页面，确认一个页面"的迭代开发方式，确保每个页面质量后再进行下一个页面的开发。

**开发流程**：

1. **页面规划阶段**：
   - 根据项目需求和风格方案，确定页面列表和优先级
   - 明确每个页面的功能需求、布局结构、交互逻辑
   - 确定页面之间的关联关系和数据流

2. **单页开发阶段**：
   - **页面创建**：创建页面文件，搭建基础结构
   - **布局实现**：根据风格库文档实现页面布局
   - **组件集成**：集成所需组件，实现页面功能
   - **样式调整**：根据风格方案调整样式，确保视觉一致性
   - **交互实现**：实现页面交互逻辑和状态管理

3. **页面自检阶段**（开发完成后必须执行）：
   - **功能检查**：验证页面功能是否完整，交互是否正常
   - **样式检查**：检查样式是否符合风格方案，响应式是否正常
   - **代码检查**：使用自动检测工具检查代码规范
   - **兼容性检查**：检查浏览器兼容性和移动端适配

4. **页面确认阶段**：
   - **提交确认**：将完成的页面提交给项目负责人或产品经理确认
   - **问题修复**：根据反馈修复问题，重新提交确认
   - **确认通过**：获得确认后，标记页面为"已完成"

5. **下一个页面开发**：
   - 只有在当前页面确认通过后，才能开始下一个页面的开发
   - 如当前页面有问题，必须先修复完成并确认通过

**流程优势**：
- **质量保证**：每个页面都经过确认，避免后期大规模返工
- **风险控制**：问题及时发现和修复，降低项目风险
- **进度可控**：清晰的进度节点，便于项目管理
- **经验积累**：每个页面的经验可以应用到后续页面

### 9.2 自动错误检测机制

#### 9.2.1 检测工具配置

**代码规范检测**：
- **ESLint**：JavaScript/TypeScript代码规范检测

**类型检测**（TypeScript项目）：
- **TypeScript编译器**：类型检查

**构建时检测**：
- **Vite构建检测**：构建时自动检测编译错误

#### 9.2.2 自动检测流程

1. **开发时检测**（实时）：
   - IDE集成：在编辑器中实时显示错误和警告
   - 保存时检测：保存文件时自动检测并提示错误
   - 自动修复：部分问题可自动修复（如格式问题）

2. **构建时检测**（必须通过）：
   - 构建前自动运行检测工具
   - 检测失败则构建失败，无法打包

#### 9.2.3 检测内容清单

| 检测项 | 检测工具 | 检测时机 | 是否阻止提交 |
|-------|---------|---------|------------|
| JavaScript/TypeScript语法错误 | ESLint | 实时、构建时 | 是 |
| 类型错误（TypeScript） | TypeScript | 实时、构建时 | 是 |
| 未使用的导入 | ESLint | 实时、构建时 | 否（警告） |
| 变量未定义 | ESLint/TypeScript | 实时、构建时 | 是 |

## 十、前端页面验收标准

> **说明**：前端页面验收标准已独立为单独文档，详细内容请参考 [验收规则.md](./验收规则.md)。

**验收标准概述**：
- 验收采用分项评分制，每项满分100分
- 最终得分 = 各项得分加权平均，目标得分 ≥ 80分
- 验收维度包括：视觉（25%）、交互（25%）、功能（20%）、性能（15%）、兼容性（10%）、代码质量（5%）、可访问性（5%）

**验收流程**：
1. **开发自检**：开发完成后，开发人员按照验收标准进行自检
2. **内部验收**：项目负责人或技术负责人进行内部验收
3. **产品验收**：产品经理或业务负责人进行功能验收
4. **最终验收**：所有验收通过后，页面标记为"验收通过"

**详细验收标准**：请查看 [验收规则.md](./验收规则.md) 文档。

# 代码不足原因分析

> **分析时间**：2025-01-30  
> **分析对象**：组卷规则弹窗缺少负数限制和合理性校验的问题

---

## 一、问题现象

**组卷规则弹窗（RuleDialog.vue）存在的问题**：
1. ❌ 数字输入框可以输入负数
2. ❌ 没有实时合理性校验提示（总和是否等于100%）
3. ❌ 保存时校验提示不够详细

---

## 二、根本原因分析

### 2.1 项目历史遗留问题（主要原因）

**代码不一致性**：

| 文件 | 数字输入处理方式 | 是否有范围限制 | 是否有事件处理 |
|------|----------------|--------------|--------------|
| `RuleDialog.vue`（之前） | 原生 `input type="number"` | ❌ 无 | ❌ 无 |
| `AutoPaperModal.vue` | 原生 `input type="number"` | ✅ 有 `min="0" max="100"` | ❌ 无事件处理 |
| `DepartmentManagement.vue` | Element Plus `el-input-number` | ✅ 有 `:min="0" :max="999"` | ✅ 组件内置 |
| `ManualPaperPage.vue` | 原生 `input type="number"` | ✅ 有 `min="1" max="100"` | ❌ 无事件处理 |

**结论**：
- 项目不同时期、不同开发者写的代码，**标准不统一**
- 有些用原生 input，有些用 Element Plus，**没有统一规范**
- 即使有 `min/max` 属性，**浏览器原生限制不够严格**（仍可输入负数，只是提交时提示）

---

  - 添加 `@input/@blur` 事件处理

- ❌ 没有**统一的数字输入组件规范**

**现有 `validators.js`**：
```javascript
export const validateNumber = (value, fieldName = '此字段') => {
  if (!/^\d+$/.test(String(value))) return `${fieldName}必须是数字`;
```

**问题**：
- ✅ 有 `validateNumber` 函数
- ❌ 但**只校验"必须是数字"**，不校验：
  - 范围（min/max）
  - 是否负数

**如果工具函数更完善**（理想状态）：
```javascript
export const validateNumberRange = (value, min, max, fieldName = '此字段') => {
```

## 三、是否由现有代码导致？

### 3.1 直接原因

**是的，部分由现有代码导致**：

1. **历史代码不一致**：
   - 不同文件使用不同的数字输入方式
   - 没有统一的校验标准
   - 导致新代码容易"遗漏"校验

2. **工具函数不完善**：
   - `validators.js` 的 `validateNumber` 功能有限
   - 没有现成的范围校验函数
   - 开发者需要自己写校验逻辑

3. **没有统一的输入组件**：
   - 原生 `input` 和 Element Plus `el-input-number` 混用
   - 原生 input 的 `min/max` 限制不够严格
   - 需要手动添加事件处理

---

### 3.2 间接原因（规范层面）

**开发规则和需求文档的问题**：

1. **开发规则不够强制**：
   - 虽然提到"范围校验"，但没有**强制要求**
   - 没有**统一的数字输入组件规范**
   - 没有**代码审查清单**（如：所有数字输入是否都有范围限制？）

2. **需求文档不完整**：
   - 需求6只说了"添加字段"，没说"要校验"
   - 没有明确"禁止负数"、"总和100%"等业务规则
   - 依赖开发者"自觉"实现校验

**建议2：完善工具函数**

在 `src/utils/validators.js` 中增加：

```javascript
/**
 * 验证数字范围
 * @param {string|number} value - 字段值
 * @param {number} min - 最小值
 * @param {number} max - 最大值
 * @param {string} fieldName - 字段名称
 * @returns {string} 错误消息
 */
export const validateNumberRange = (value, min, max, fieldName = '此字段') => {
  if (value === '' || value === null || value === undefined) return '';
  const num = Number(value);
  if (isNaN(num)) return `${fieldName}必须是数字`;
  if (num < min) return `${fieldName}不能小于${min}`;
  if (num > max) return `${fieldName}不能大于${max}`;
  return '';
};

/**
 * 处理数字输入（禁止负数，限制范围）
 * @param {string} field - 字段名
 * @param {Event} event - 输入事件
 * @param {Object} form - 表单对象
 * @param {number} min - 最小值（默认0）
 * @param {number} max - 最大值（默认100）
 */
export const handleNumberInput = (field, event, form, min = 0, max = 100) => {
  const target = event.target;
  let value = target.value;
  
  if (value.startsWith('-')) {
    value = value.replace('-', '');
  }
  
  const numValue = Number(value);
  
  if (value === '' || isNaN(numValue)) {
    form[field] = undefined;
    return;
  }
  
  if (numValue < min) {
    form[field] = min;
  } else if (numValue > max) {
    form[field] = max;
  } else {
    form[field] = numValue;
  }
};
```

## 五、其他可能存在类似问题的模块

除组卷规则弹窗外，项目中还有多处数字输入，**未统一做范围与事件处理**，存在同类风险。下表为全项目排查结果（基于当前代码）。

| 模块/文件 | 数字输入用途 | 当前状态 | 建议 |
|-----------|--------------|----------|------|
| `RuleDialog.vue` | 达标分数（passingScore） | 无 min/max，无 @input/@blur | 补全 0–100 范围及事件处理，与题型比例一致 |
| `RuleDialog.vue` | 题型比例（workEthics/base/expert） | 已有 min/max 与事件处理 | 保持，可考虑抽成组件复用 |
| `PaperManagement.vue` / `PaperManagement2.vue` | 考试时长（duration） | 仅 `type="number"`，无 min/max/事件 | 增加合理范围（如 1–600 分钟）及事件处理 |
| `AutoPaperModal.vue` / `AutoPaperManage.vue` | 题型数量、单题分值 | 部分有 min/max，无 @input/@blur | 统一加事件处理，禁止负数与超范围 |
| `ManualPaperPage.vue` | 题目数量 | 有 min="1" max="100" | 建议增加 @input/@blur 防止负数与越界 |
| `StudentDetail.vue` / `StudentScore.vue` | 题目得分 | `type="number"`，部分无范围 | 按题目满分限制范围，并禁止负数 |
| `DepartmentManagement.vue` | 部门排序（deptOrder） | 使用 `el-input-number`，:min/:max 已配置 | 已规范，无需改动 |
| `examStation/index.vue`、`ExamManagement.vue`、`ExamApplicationReport.vue` 等 | 各类数量/分数 | 多为原生 number，无统一限制 | 按业务补充 min/max 与事件处理 |

**结论**：  
- 多数数字输入仍为**原生 `input type="number"`**，仅少数使用 `el-input-number`。  
- **达标分数、考试时长、得分**等关键字段若不做范围与事件约束，仍可能出现负数或不合理值。  
- 建议按「附录：数字输入整改清单」逐项收敛，并优先处理与组卷、发卷、成绩相关的页面。

## 八、总结

### 8.1 主要原因排序

1. **项目历史遗留**（60%）：代码标准不统一，不同文件处理方式不同
2. **开发规则不够具体**（25%）：虽然提到校验，但没有强制要求和统一规范
3. **需求文档不完整**（10%）：需求6没有明确校验要求
4. **工具函数不完善**（5%）：现有校验函数功能有限

### 8.2 是否由现有代码导致？

**部分是的**：
- ✅ 历史代码的不一致性，导致新代码容易"遗漏"校验
- ✅ 工具函数不完善，需要开发者自己写校验逻辑
- ❌ 但**主要责任不在现有代码**，而在：
  - **开发规范不够强制**
  - **需求文档不够详细**
  - **没有统一的组件规范**

### 8.3 如何避免类似问题？

1. **完善开发规则**：强制要求所有数字输入必须有范围限制和事件处理
2. **完善需求文档**：明确所有校验要求（禁止负数、范围、实时提示等）
3. **创建统一组件**：封装数字输入组件，统一处理校验
4. **代码审查清单**：提交前检查是否满足校验要求
